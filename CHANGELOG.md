# Changelog

A subsystem-level overview of what's been built. Organized by area rather than date — the history is in git.

## Site scaffold

Next.js 16 + React 19 + Tailwind CSS v4, deployed to Vercel. App Router with per-route SSG where possible; dynamic API routes on the Node runtime. LocalBusiness JSON-LD is wired into the root layout from `siteSettings` so the `<head>` of every page carries structured data. `robots.ts` and `sitemap.ts` are auto-derived from the visible service + portfolio getters.

A tab-style **section sub-nav** (`SubNav`) sits above the page title on every page in the public page groups, mirroring the admin sub-nav's look (uppercase tracking, a bottom rule) with the active tab matched by pathname and rendered in the gold accent. The **About Julian | Journal** group spans the about page and every journal post (a post highlights _Journal_), and the top-level **Portfolio** and **Services & Pricing** indexes link across to each other. A **category detail page**, though, scopes its tabs to that one category: `/portfolio/weddings` and `/services/weddings` share a "Weddings portfolio | Weddings pricing" pair (the sibling tab appears only when that sibling is published) plus an inline **back link** up to the index, set off from the tabs by a thin divider. That single affordance supersedes both the old per-page "← back" link and the old top-right "view pricing / view portfolio" cross-link, so each category offers exactly one way to switch facets and one way back. Two action groups round it out: the **booking funnel** — Inquire | Plan your session | Book | FAQ — spans `/inquire`, the planning questionnaire, `/book`, and the FAQ directory; the **client surfaces** — Client portal | Client galleries — sit on the portal and the galleries page. "Plan your session" lives only in the funnel, so no page belongs to two bars; the portal's project pages link to the questionnaire contextually instead. The same `SubNav` also powers the authenticated portals — the admin and client areas both render it with an integrated **Sign out** (its `logoutAction`), so there's one nav component site-wide and the two portals read identically. The main nav's Clients dropdown carries the Client portal link. The portal sign-in page (`/portal`) carries the client bar and forwards an already-signed-in visitor to their dashboard so its "Client portal" tab resolves from either state; the `/admin` sign-in page stays tab-free. Every sub-nav-bearing page shares one wide nav region (a `max-w-7xl` row at a uniform `py-20` top offset — public, portal, and admin alike, so the bar sits at the same height everywhere) with narrower content — a login form, a questionnaire — inner-wrapped beneath it, so the rule's length and vertical position match on every page.

Three content pages put their secondary panel in a **sticky second column** on desktop, collapsing to one column on mobile: `/inquire` pins the "already booked?" planning callout beside the form, `/faq` pins the concierge chat beside the searchable directory, and `/about` pins a photo column beside the bio. The about photos are path-based like the portfolio galleries — a source `about/` folder runs through the import script into `public/about/` and a generated `about-manifest.ts` the about page merges — so adding them is one command, and the column simply doesn't render until they exist. And `/faq` no longer yanks the viewport down to the chat on load: the concierge scrolls its own log container, not the page.

Page-ending calls to action share one **`CalloutCard`** — an accent-bordered white card with an optional uppercase eyebrow, a serif title, an optional description, and pill action buttons, modeled on the service pages' "Booked or seriously considering?" box. The loose button clusters and differently-styled boxes that used to live on `/about`, `/client`, `/inquire`, `/book`, the portal sign-in page, and the services + questionnaire indexes now all render through it (the old top-right "View Services / Portfolios" index cross-links are retired in favor of the section nav), so every CTA reads as one UI.

The pill buttons inside `CalloutCard` (and the standalone CTAs that aren't in one) come from a single **`Button`** primitive — one place owns the variants (primary / secondary / ghost) and the size scale (sm / md / lg), so the padding can't drift (`px-5 py-2` vs `px-6 py-3` had crept across the site) and a CTA always reads the same. Interactive form buttons (submit/save with their own disabled + loading states) stay inline by design. Color stays anchored to the five-token palette (`--background`, `--foreground`, `--muted`, `--accent`, `--border`) defined in `globals.css`: a couple of off-palette grey status badges that had slipped in (`stone-*`) were re-pointed at the brand neutrals, leaving only semantic red/amber for urgency — the one place a non-brand hue is intentional.

While a route segment waits on data, the App Router shows one shared **loading identity** — a centered line-art camera (`LoadingScreen`) above a "Loading…" label, wired as the root `app/loading.tsx` Suspense fallback. It surfaces only when a navigation genuinely suspends — the dynamic admin/portal Supabase reads, a cold Sanity fetch — and stays invisible on instant or prerendered pages, so the few per-route journal/portfolio skeletons retired into it and every wait across the site now reads the same.

## Project structure & conventions

`src/components/` is organized by domain. **`ui/`** holds the generic primitives — `Button`, `AiButton`, `Panel`, `CalloutCard`, `RailCard`, `CopyField`, `SubNav`, `LoadingScreen`, `SafeImage`, and the `fields/` form controls. **`forms/`** holds the public-facing forms (inquiry, the questionnaire with its `questionnaire/` renderer internals, both new-project forms, `AssistedTextarea`). **`admin/`** holds the owner-dashboard pieces, with the AI drafting tools grouped under `admin/ai/`. **`portal/`** holds the client-portal pieces, and **`marketing/`** the public site chrome and content components (the nav with its `nav/` menu internals, footer, galleries, concierge, FAQ browser). `src/lib/` keeps the data/content modules flat and clusters the rest: **`lib/email/`** (pipeline templates, branded layouts, send helpers, notifications), **`lib/ai/`** (model client, voice, concierge knowledge base), and **`lib/pdf/`** (the printable plan documents).

Repeated UI resolves to one source. Every form control renders through the `fields/` primitives on the shared `Field` chrome — including **`SelectField`** for dropdowns (options as plain strings or value/label pairs, a placeholder option that locks for required choices or stays selectable for clearable ones, and a trailing slot for a conditional "other" input) — and the styling constants live beside them: `inputClass` / `controlClass` / `labelClass` for full-size controls and `compactInputClass` for dense tool rows (the admin quick-log, the list filter bar, the portal upload row). Inline form-submit pills that manage their own loading state reference `submitButtonClass`, exported next to `Button`; purely navigational pill CTAs render through `Button` itself. Content boxes are three cards: **`Panel`** (the neutral white box), `CalloutCard` (accent-bordered CTA), and `RailCard` (labeled rail module). The one human-readable date formatter is `formatHumanDate` in `src/lib/field-format.ts` — "June 7, 2026", or "Jun 7, 2026" with the short month for email subjects — shared by the journal, the portal and admin pages, and the email pipeline — joined by `formatRelativeDays` ("3 days ago") and a `STUDIO_TIME_ZONE` pin for displaying stored timestamps. Record-field display labels (source, admin status titles, document types, uploader names) live in `src/lib/labels.ts`, so a stored slug never reaches the screen; conversions with an existing domain home stay there (service titles in `services-data.ts`, referral labels in `referral.ts`, client-facing status phrasing in `client-status.ts`).

The largest components are split into focused units that their shells compose: the questionnaire's per-field rendering (`FieldRenderer`), uploads (`FileField`), and referral dropdown live in `forms/questionnaire/`; the admin form's venue/address list is `admin/AdminLocationsEditor`; and the nav's menu data, hover panels, and mobile drawer live in `marketing/nav/`, so the desktop megamenus and the mobile drawer read from one menu definition.

## Form fields & formatting

Every form on the site — the inquiry form, the multi-section questionnaire, the admin edit form, portal edits, and both new-project forms — runs its inputs through one set of shared field primitives (`src/components/ui/fields/`) rather than hand-rolled `<input>`s, so formatting, validation, and the label / required-marker / inline-error chrome read identically everywhere (all off one `Field` wrapper + `inputClass`, with the helpers in `src/lib/field-format.ts`). **`PhoneField`** masks to `(703) 555-1234` as you type — the visual companion to `normalizeE164`'s server-side E.164 pass; **`EmailField`** flags an invalid address on blur and offers a one-click fix for a common domain typo ("Did you mean name@gmail.com?"); **`BudgetField`** drops in a `$` and comma-groups a plain number; **`NumberField`** holds to non-negative integers; and **`DateField`** types as `MM/DD/YYYY` (auto-slashing, auto-advancing month→day→year) and opens a **react-day-picker** calendar styled as a callout-card popover — with month + year dropdowns (a far-out date is one click, not a dozen arrow taps) and the selected day ringed in the brand accent — while storing ISO `yyyy-MM-dd`.

**Address & venue autocomplete.** `LocationField` queries `/api/places/autocomplete`, a server proxy for Google Places (New) that reuses the existing `GOOGLE_PLACES_API_KEY` (the reviews key), builds lightweight predictions from each suggestion's `structuredFormat` (no Place Details call, so no extra per-select billing), is rate-limited, and **returns an empty list when the key is unset** so the field cleanly degrades to free text. As the user types (≥3 chars, debounced) a keyboard-navigable combobox of addresses/venues drops in; selecting one fills the field, and **free text is always retained** for custom entries. It powers the inquiry "location", the questionnaire venue/address fields (`venueName` / `venueAddress` / get-ready / reception), and a new editable **Locations** section on the admin project form — a repeatable label/address/notes list persisted to the `locations` JSONB through `/api/admin/update`.

## Inquiry form + email

`/inquire` posts to `/api/inquire`. Resend sends the owner a structured inquiry email plus an auto-reply to the client. The handler writes the inquiry subject line with the event date formatted as `"Aug 15, 2027"`. The form marks its required fields — name, email, service, and the message — with an asterisk and validates them client-side on submit: a missing or malformed field shows an inline, `aria-describedby`-wired error and focuses the first offender instead of bouncing off a generic server message.

**Every transactional email shares one branded layout.** The inquiry + auto-reply, questionnaire submissions, the portal and admin magic-link sign-in links, and the client-update notification to Julian all render through `BrandedEmailLayout` in `src/lib/email/email-templates.tsx` (React Email → HTML via `render()`, with a plain-text fallback always included) — so the login links and notifications match the inquiry/questionnaire look (serif header, accent rule, footer with contact + socials) instead of ad-hoc inline HTML. Reusable content templates: `MagicLinkEmailTemplate` (portal/admin sign-in) and `NotificationEmailTemplate` (generic heading + lines + optional CTA, ready for the planned client/admin notifications). Every send draws its `from` from one `resendFrom()` helper (`src/lib/email/email-helpers.ts`) that defaults to the studio's own domain (`noreply@julianperezphotography.com`, overridable via `RESEND_FROM`) instead of the shared `resend.dev` sandbox sender, and each client-facing message carries a `replyTo` pointed at Julian's real inbox — so replies reach him without a domain mailbox to host while sending stays on the verified domain. `resendFrom()` warns loudly if it ever resolves to the `onboarding@resend.dev` sandbox sender, which silently delivers only to the Resend account's own address (every other recipient is 403-rejected) — so a stray `RESEND_FROM` override can't quietly break client mail.

Honeypot + in-memory per-IP rate limiting (`src/lib/request-guard.ts`) guard against basic abuse — 5 submissions per 10 minutes per IP, keyed per endpoint so the inquiry form and questionnaire limiters don't share a bucket.

When `RESEND_API_KEY` is unset the handler logs the payload and returns `{ ok: true, dev: true }` so local development works without a Resend account.

## SMS confirmations (optional)

When all three `TWILIO_ACCOUNT_SID` / `TWILIO_AUTH_TOKEN` / `TWILIO_PHONE_FROM` env vars are set, `/api/inquire` also dispatches a Twilio SMS confirmation. Strictly additive to the email flow: SMS is wrapped in try/catch so a Twilio failure never surfaces into the user-facing response. Phone-number normalization (`(703) 555-1234` → `+17035551234`) is a pragmatic DMV-focused pass — Twilio itself rejects malformed numbers server-side.

## Questionnaire system

Schema-driven planning forms at `/questionnaire/[service]`, defined by the `Questionnaire → Section → Field` tree in `src/lib/questionnaires.ts`. Field types: `text`, `textarea`, `email`, `tel`, `date`, `time`, `number`, `select`, `radio`, `checkbox`, and `package` (resolves at render-time from `services[slug].packages` so package lists never drift from pricing).

- Drafts autosave to `localStorage` and clear on successful submit.
- URL query-params prefill fields so clients can land on a partly-filled form from a post-booking email link.
- The wedding form's **sunset session time auto-defaults** to the ideal golden-hour start. Once the event date and a venue are filled in, a client-side effect calls `/api/sun`, which geocodes the venue via Places Text Search (New) (`geocodeAddress` in `src/lib/places.ts`, reusing `GOOGLE_PLACES_API_KEY`), looks up its IANA timezone (`tz-lookup`), computes the real sunset there (`suncalc`), and returns a start time ~25 min before it in the venue's own local clock. It only fills an empty field while the question is shown, so a manual choice is never overwritten, and soft-fails to no prefill whenever the venue can't be resolved.
- Conditional `showIf` branches hide/show follow-up fields; the server validates against the same schema so anything hidden on the client is also optional on the server. `showIf` matches against checkbox (array) values by membership, which lets a checked option reveal its own follow-up fields — the wedding questionnaire's "Vendors & coordination" section uses this so checking a vendor type opens a contact box for it.
- The wedding questionnaire captures photography style preference (with an "Other" free-text branch), engagement-session preferences (shown only for packages that include one), a structured **ceremony type** (religious / civil / cultural / hybrid) and **cultural-traditions** prompt, expected ceremony length, an outdoor-ceremony weather-backup plan, vendor contacts by type, and photo-sharing consent (full / none / per-channel-and-type partial). The ceremony-type and cultural questions mirror the wedding-films questionnaire so a hybrid couple sees the same questions on both forms.
- `/api/wedding-plan` renders a PDF preview of the wedding questionnaire for editor / client review.

## Webhook revalidation

`/api/sanity-webhook` receives signed POSTs from Sanity on every publish, verifies the HMAC via `@sanity/webhook@4`'s `isValidSignature` against the raw request body, and invalidates **both** cache layers:

1. Next's Data Cache — `revalidateTag(tag, { expire: 0 })` on the collection tag plus any per-slug tag, so the next render refetches from Sanity.
2. Vercel's edge CDN — `revalidatePath(path)` on each route that renders the changed content, so the next request is a cold miss instead of serving `s-maxage=60` HTML.

Without the path-level call, the webhook returned 200 but the site kept serving stale HTML for up to 60s. Both calls together deliver sub-second end-to-end propagation. `siteSettings` is special-cased to `revalidatePath("/", "layout")` because the root layout reads it on every route.

Signature-verified means an attacker with just the endpoint URL can trigger 401s at worst, never force a revalidation. Unknown `_type` values 400-and-log so a new schema added in Studio without the corresponding handler update surfaces loudly in the Sanity webhook attempt log instead of silently no-op'ing.

## Cross-questionnaire prefill (service bundles)

A client booking more than one service — a couple's wedding photo + film + engagement session, or a growing family's maternity → newborn → family portraits — shouldn't retype the answers two forms share. The success screen on any questionnaire offers a "Continue planning your {sibling} →" button for each other service in its **bundle group**, and the destination form arrives prefilled.

The mechanism is **intersection-based**, so it scales to any bundle with no per-pair field lists: each sibling button's URL carries exactly the answers the *target* form also asks for — the intersection of what was answered and `prefillableFieldIds(target)` (every field except file uploads, which can't ride a query string). Multi-select (checkbox) answers ride along as repeated query keys, so they round-trip in full — not just single-value fields. The receiving page turns query params into prefill (preserving arrays), so nothing else changes on the receiving side. Bundle groups are a one-line curated config (today `{weddings, wedding-films, engagements-couples}` and `{maternity, newborn, family-portraits, family-celebrations}`); adding a future bundle is adding a slug.

Beyond the post-submission cross-prefill, every applicable questionnaire now **asks about bundling in-form, consistently**. `getQuestionnaire` appends a generated **"Related sessions"** section — one checkbox listing the OTHER services in the slug's bundle group (resolved to their service titles), worded identically on every form — so a client's interest becomes a recorded answer Julian sees in the questionnaire read-out, not just a navigation link offered afterward. The bundle groups moved into `questionnaires.ts` so the same data drives both the prompt and the cross-prefill (the generated `bundleInterest` field is excluded from cross-prefill, since its options differ per service). Services with no questionnaire-backed siblings (graduation, headshots, concerts, …) never get the section, so it appears only where applicable.

What makes the carry-over actually work is a one-pass **field-ID harmonization across all 15 questionnaires**, which had drifted as they were authored in separate passes. The same concept now uses one canonical ID everywhere: partner's name (`partnerFullName`), the vibe/mood field (`vibe`), wardrobe help (`wardrobeHelp`), how-you-met (`howWeMet`), time-of-day (`timeOfDay`), pets (`pets`), and the shared wedding vendor contacts (`plannerInfo`, `venueCoordinatorInfo`) — on top of the ceremony block harmonized earlier. **Venue is unified** to a canonical `venueName` + `venueAddress` split on every form with a known event venue (weddings keeps a second "reception venue, if different"); forms that only collect pre-booking *location ideas* (`locationIdeas`) stay as-is, since that's a different question. Only the two wedding PDF plan modules hard-code field IDs, so they were updated to match; every other rename is internal to the schema. Concepts that merely *look* shared were deliberately left split — `partner` ("will they be in the photos?") is a yes/no, not a name, and weddings' `specialDances` (which dance moments to photograph) vs. the film's `musicLove` / `musicAvoid` (soundtrack taste) are genuinely different questions.

The success screen's layout was also tidied: the **"Track your project" portal CTA is the primary (dark) button** above the bundle card, and literal `→` / `—` escapes that rendered verbatim in button labels were corrected to glyphs.

## Observability (Sentry)

`@sentry/nextjs` is wired into the project per the upstream SDK's recommended layout. Four entry-points cover every runtime: `instrumentation.ts` at the project root dispatches to the server / edge configs through Next's `register()` hook and exports `onRequestError = Sentry.captureRequestError` so React Server Component + route-handler failures land in Sentry without per-route boilerplate; `instrumentation-client.ts` initializes the browser SDK and exports `onRouterTransitionStart = Sentry.captureRouterTransitionStart` so App Router navigations show up as performance spans; `sentry.server.config.ts` and `sentry.edge.config.ts` carry the runtime-specific `Sentry.init` calls. All four are no-ops when `NEXT_PUBLIC_SENTRY_DSN` (or the server-side `SENTRY_DSN`) is unset, so local dev without a DSN runs unchanged.

`next.config.ts` is wrapped in `withSentryConfig` for build-time source-map upload (gated on `SENTRY_AUTH_TOKEN` so non-CI builds skip the upload step; `widenClientFileUpload: true` covers dynamic-import chunks; maps are deleted from the deploy output after upload to defend against bundle leakage) and a same-origin tunnel route (`/monitoring/sentry`) that bypasses ad-blocker drops on client-side error reporting.

Browser-side Session Replay is enabled at 10% ambient capture + 100% on-error capture, with `maskAllText` / `maskAllInputs` / `blockAllMedia` pinned explicitly so the questionnaire / inquiry forms' real client PII can't end up in recorded sessions. Server config has `includeLocalVariables: true` (variable values attached to stack frames for diagnosis without a repro) and `enableLogs: true` (activates Sentry's Logs product alongside captured exceptions).

`Sentry.captureException` calls land inside every existing API-route catch block (inquire, questionnaire, wedding-plan, wedding-films-plan, questionnaire-upload) plus the App Router error boundaries (`error.tsx`, `global-error.tsx`). Each call carries `tags: { route, stage }` so events group cleanly in the Sentry dashboard. Fire-and-forget side-effects (SMS, client confirmations) capture at `level: "warning"` to keep the primary failure dashboard signal-heavy.

A `beforeSend` hook strips request body / headers / cookies from every captured event before send, leaving only the URL and method — questionnaire and inquiry payloads carry real client PII that should never leave the project's own infrastructure.

CSP additions: `connect-src` += `*.sentry.io` + `*.ingest.sentry.io` (defensive; the same-origin tunnel route is the primary path), new `worker-src 'self' blob:` directive (Sentry's profiling features use Web Workers).

A `# Test + observability discipline` section in `AGENTS.md` codifies the standing requirement: new code paths ship with a smoke-level e2e test and Sentry instrumentation alongside the code.

## Client records & CRM

Inquiries and questionnaire submissions are now captured as durable **client records** (closing the long-standing submission-archive gap) in a free, **private Supabase Postgres** table (`client_records`). Sanity isn't used for this: client records carry PII, a private store is required, and Sanity's private datasets are a paid feature — a free Postgres table keeps the data private at no cost. Julian manages records in the owner dashboard at `/admin` (see "Owner admin dashboard"); the Supabase Table Editor remains available for complex JSONB edits.

The app reaches the table only through `src/lib/clients.ts` (server-only, Supabase service-role key). Reads use a few projections: the full row for admin/server reads and `SAFE_SELECT` for the portal, which omits `internal_notes`, status history, and inquiry context at the query layer so they can never reach a client. The questionnaire snapshot is held out of the portal **list** the same way, but the single-project owner read (`getProjectForEmail`) layers it back on so a signed-in client can review their **own** answers — gated by project id + session email, so it never reaches the dashboard list or anyone else's project. Row-Level Security is enabled on the table; the service role bypasses it server-side and nothing else can read it. The canonical pipeline statuses live in a dependency-free `src/lib/client-status.ts` shared by the helpers and the portal. Array fields (locations, dates, documents, status history) are JSONB columns.

**One row is one project; the email identifies the person.** The email column is non-unique, so a single client can own several projects at once — an engagement and a wedding, a maternity and a newborn session, or unrelated shoots. Projects that belong together share a `bundle_id` with a denormalized `bundle_label` (see "Project bundles"). Helpers resolve a person's projects by email (`listProjectsByEmail`, `getProjectForEmail`) and gate every read/write on ownership.

Beyond inquiries and questionnaires, **projects can be created directly** on both sides. Julian creates one from `/admin/projects` (`/api/admin/create-project` → `createProjectManual`) with whatever he has — even just an email and "Not sure yet" for the service; a returning client starts one from their portal dashboard (`/api/portal/create-project`, email taken from the session, never the body). Both **warn on a duplicate** — an existing email+service project found by `findDuplicateProject` — but allow a second ("create another anyway"). Created projects expose a **completion link** that threads the project id: the admin project page surfaces a copyable client link (`/questionnaire/<service>?project=<id>&…` when the service is set, `/inquire?project=<id>&…` when it's undecided), and the portal's own questionnaire/inquiry links carry it too. When that submission lands, `upsertClientFromInquiry` / `attachQuestionnaire` attach to **that exact project** — filling blanks, claiming the email/name, and setting the service if it was undecided — instead of spawning a duplicate, **but only when the project is unclaimed or its email matches the submitter**. A UUID id + the email-match guard + blanks-only fill mean a leaked link can't overwrite someone else's curated record, which keeps the project param safe on the public `/inquire` route. Net-new people with no record still begin at `/inquire` (the portal sign-in page routes them there), so it stays the single vetted front door.

Every project shows a friendly **display name** rather than the bare service slug — a custom one the client or admin set (the optional `project_name` column, editable from either side), or otherwise an auto name assembled from who + what + when, e.g. "Julian Perez's Wedding on 6/7" or "Carolyn Schook's Maternity Session on 5/9". The naming logic is a pure helper in `src/lib/project-name.ts` (`projectDisplayName` / `autoProjectName`) shared by the portal and admin. The **service** likewise reads as its catalog title everywhere (`serviceTitle` in `src/lib/services-data.ts`) rather than a raw slug, and the **package** is one canonical value — the bare tier name — across the questionnaire answer, the record, the admin picker (which no longer appends the price), and both portal displays.

The read layer tolerates schema drift: if a deploy lands before a migration's columns are added (the post-launch optional columns are the bundle id/label, the gallery URL, and the custom project name), the first projection that asks for them catches PostgREST's undefined-column error (SQLSTATE 42703), drops the optional columns for the rest of the process, and retries — so the portal and admin keep serving (minus those fields) instead of returning a 500. An unrelated error is matched out and still surfaces. Applying the migration and restarting restores them.

Capture is fire-and-forget and **no-ops when the store isn't configured**, so the email flow is untouched on a deploy without the Supabase env. `/api/inquire` upserts a record (matched by normalized email **+ service type**, so a second service for the same person opens a second project instead of overwriting the first; status `new-inquiry`; never clobbers an advanced record). `/api/questionnaire` snapshots the answers onto the matching project (email + service), **syncs the structured answers it captures — package, event date, guest count, partner name — onto the record's own columns** so the portal and admin reflect what the client entered without re-keying, stores the generated plan PDF to Vercel Blob (`src/lib/blob.ts`), links it as a document, and advances status toward `planning` without regressing a further-along record.

## Client portal

A passwordless client portal at `/portal`. Magic-link auth via `jose`: `/api/portal/request-link` emails a one-time signed token (20-min TTL) through Resend with a uniform, rate-limited, anti-enumeration response; `/portal/verify` validates it, confirms at least one project exists for the email, and sets an httpOnly/Secure/SameSite session cookie carrying the **email** (the person), not a single record id. The gate is a Next.js 16 **Proxy** (`src/proxy.ts`, Node.js runtime) that protects `/portal/*` except the login + verify routes — running on the same runtime as the page renderers keeps cookie handling consistent across full-page loads and client-side navigations. Token + session signing share `AUTH_SECRET`; the pure `jose` sign/verify helpers live in `auth.ts` (used by both the proxy and the route handlers), while the `next/headers` cookie read/write helpers live in `auth-cookies.ts`. Sign-out (`/portal/logout`, `/admin/logout`) is a **`POST`** triggered by a form button, never a prefetchable `GET` link — so a link prefetch, which is always a GET, can't clear the session out from under the user.

The portal opens to a **menu of the signed-in person's projects** — styled like the services / galleries index, with cards grouped under Julian's photographic umbrellas and bundled projects pulled into their own accent-bordered group. `/portal/projects/[id]` then renders one project's detail via the `SAFE_SELECT` projection: friendly status label, dates, locations, plan, package/service, and document download links. The detail leads with a **milestone timeline** — the twelve pipeline statuses collapse into six client-facing stages (Inquiry → Booked → Planning → Shoot day → Editing → Delivered, in `client-status.ts`) rendered as a stepper with a warm "what's next" hint and, when the event date is known and upcoming, a friendly countdown; the menu cards carry a compact progress bar of the same milestones. When Julian sets a **gallery URL** on a project, the detail surfaces a prominent "Your gallery is ready" card linking out to the Pic-Time delivery. Every project is resolved by **(project id + session email)** — a URL naming someone else's project returns not-found, so listing and linking can never reach across people (no IDOR). On a project, clients can make **limited edits** (`/api/portal/update` — a zod-whitelisted set: phone, partner name, guest count, and their own notes / questions) and **upload documents** (`/api/portal/upload` issues a session-gated Blob token; `/api/portal/attach-document` links the upload to that project with `uploadedBy: "client"` and a client-chosen **type** — general / contract / invoice / timeline / moodboard — shown as a badge); both carry the `projectId` and re-check ownership server-side. Julian is emailed on client edits. All portal routes are Sentry-instrumented; `/portal` is disallowed in `robots.ts`. The project detail view carries the unified portal nav (the section bar + an integrated Sign out) and lays out full-width in two responsive columns on wide screens, split cleanly by purpose — **client updates & submissions** in the main column (the edit form, document upload, and an inquiry prompt when no service is set yet) and **static project info** in a side rail (an enriched **At a glance** card, the read-only Plan, the planning-questionnaire callout, documents) — collapsing to one column on narrow screens. **At a glance** consolidates the project facts that apply — service, package, event date, guest count, partner, budget — with **Other dates** and a multi-entry **Locations** sub-section folded in, so the reference info reads as one card instead of several. The **planning questionnaire** sits in that rail as a collapsible, state-aware callout (shared with the admin page): once submitted it collapses to a "✓ Submitted" card that expands to the client's own section-grouped answers with a **Resubmit to update** action — the project-threaded questionnaire link, which overwrites the snapshot and re-syncs the structured fields on submit — and before then it's a plain "open your questionnaire" prompt card (the shared `CalloutCard`), so a project without a submitted questionnaire still has an obvious way in.

The shoot **Plan** and the client's **notes / questions** are now distinct fields rather than sharing one column. The Plan (`plan_summary`) is Julian's to author and the client sees it read-only; the client writes their own notes / questions (`client_notes`), and Julian can post a **reply** (`client_notes_reply`) that surfaces back in the client's portal beside their note — and, when he opts to notify, reads as "I replied to your notes / questions" in the update email. Both new columns are migration-added and read through the same degrade-gracefully guard as the bundle / gallery columns.

Each project's detail view also surfaces a **prefilled link into the matching planning questionnaire** (`getQuestionnaire(serviceType)`), carrying over name/email/phone/partner/date/guests — so a client who only inquired can flow into the full questionnaire, which attaches back to the same project (matched by email + service) and populates it further. Portal links are surfaced at every relevant touchpoint: the inquiry and questionnaire thank-you panels, the footer's "For clients" list, and the `/client` galleries page. The `/client` page itself leads with the Pic-Time embed (given a little more height) and closes with the "Galleries & project portal" access callout below the reviews, so the gallery is the first thing a returning client sees.

## Owner admin dashboard

An owner-only dashboard at `/admin`, gated by a magic link to `ADMIN_EMAIL` (the same `jose` mechanism as the client portal, with a separate session cookie + purpose; the allowlist is re-checked on every verify, so dropping an address revokes access even on an unexpired token). The Proxy (`src/proxy.ts`) gates `/admin/*` alongside `/portal/*`. The login form is shared with the portal (`PortalLoginForm`, parameterized by endpoint).

`/admin/projects` is a status-grouped overview of every client record (cards laid out like the services/galleries pages — New inquiries / Active / Delivered / Archived, each card showing a bundle badge when the project is linked, plus an inline **quick-update** control to change status or jot a status-history note via `/api/admin/quick-log` → `addAdminLog` without opening the record; a **search + filter bar** narrows the board server-side by name/email/service text and by status / service dropdowns), and `/admin/projects/[id]` is a full read/edit view: a form for the scalar fields + status dropdown + internal notes + a **gallery URL** (the Pic-Time delivery link, surfaced in the client portal) — writing through `/api/admin/update`, which appends a status-history entry on status changes — a **bundle control** that links or unlinks this client's related projects (shown when the person has two or more, writing through `/api/admin/bundle`), plus read-only context (locations, dates, documents, status history, and the planning questionnaire rendered through the **same collapsible callout the client sees** — collapsed by default, expanding to a section-grouped read-out of label/value rows resolved from the questionnaire schema via `buildAnswerGroups`, with the raw JSON behind its own disclosure rather than dumped on the page). A project can also be **permanently deleted** from a type-to-confirm danger zone (`/api/admin/delete-project` → `deleteClient`). The detail page lays out full-width in two responsive columns — a main editing column with labeled sections — **Project details** (the edit form), **Inquiry** (with its AI triage), **Questionnaire** (the collapsible callout, AI prep brief just after), and **Compose email** — each AI panel a white card so it reads as a distinct module against the page, and a context rail (suggested next step, the client completion link as a short hyperlink + Copy, details, bundle, danger zone) — so each AI helper sits with the data it acts on, and the whole thing collapses to one column on narrow screens. When a client has written notes or questions, they surface read-only on the edit form with a reply box that writes `client_notes_reply` (shown back to the client in their portal) — kept separate from the admin-authored, client-visible **Plan** and the private internal notes. The edit form is **typed to prevent drift**: the service is a dropdown of the catalog's slugs, the package is a dropdown scoped to the selected service's tiers (and clears when the service changes, so the two can never contradict), the event date uses a native date picker, and only genuinely free-form fields (name, partner, notes, plan, budget) stay as text — a legacy date that predates the picker degrades to a text field so its value is never dropped. This replaces the Supabase Table Editor for day-to-day management; the table editor remains for complex JSONB edits. `/admin/links` is a categorized hub of every external service the site depends on (Vercel, Sanity manage + Studio, Supabase, Resend, Twilio, Groq, Google Cloud, Square, Pic-Time, Sentry, GitHub, Search Console, and more). Admin reads use the full row; `listClients` / `getClientFull` / `updateClientAdmin` live alongside the client helpers in `src/lib/clients.ts`. `/admin` is disallowed in `robots.ts`.

A **needs-attention** strip at the top of `/admin/projects` surfaces stale inquiries (a `new-inquiry` untouched for 3+ days) and upcoming shoots (an event date within 30 days, still active), computed server-side from the full list. And editing a project — or quick-logging a status change — carries an opt-in **"email the client"** checkbox: when ticked it sends the client a branded `NotificationEmailTemplate` that **lists what actually changed**. A `summarizeClientChanges` diff of the pre-update snapshot against the submitted fields renders the moved values (status, event date, package, guest count, plan, a newly-ready gallery) as an accent callout list, with a portal link — while the internal note, budget, and raw service slug are deliberately excluded, and an edit that touches none of those falls back to a generic "there's an update". It sends through the shared `sendClientUpdateEmail` helper in `src/lib/email/notify.ts`. The reciprocal "a client edited their details" email to Julian is on by default and muted with `NOTIFY_CLIENT_EDITS=false`. Every send is fire-and-forget via Resend (`level: "warning"` to Sentry) and no-ops when Resend is unconfigured.

The detail rail's **Details card** renders stored values as friendly text rather than raw data: the project's **source** reads as words ("Inquiry form", "Questionnaire", "Added manually", "Started in the portal"), document meta shows the type and uploader labels, the **status history** is a newest-first list of pipeline titles with notes and human dates, and **Created / Last updated** read as "June 7, 2026 · 3 days ago" — dates pinned to the studio timezone so a late-evening record doesn't display as the next day.

Every project also carries a **compose-email** panel that turns the communication pipeline into one-click drafts. A stage-grouped picker (`<optgroup>` per pipeline stage) offers starter templates spanning the whole arc — from the first inquiry reply, a "not available" off-ramp, and the discovery-call invite/recap, through proposals and the hold-the-date and follow-up, the booking welcome, contract/deposit and payment reminders, planning kickoff and questionnaire nudges, the day-of timeline and week-of note, post-shoot thank-you and sneak peek, gallery delivery and prints/albums, to the review and referral asks, anniversary, a cooled-lead re-engagement, and a gracious **booked-elsewhere** goodbye for a prospect who picked another photographer (`src/lib/email/email-pipeline.ts`) — each prefilling from the project: first name, display name, service noun, formatted event date, venue, and the project's gallery / portal / booking / questionnaire links. Tokens without a value, and free-form `[bracket]` prompts, are left visible so the gaps to fill are obvious before sending. The owner edits freely, then **Send** delivers the message to the client through the shared branded layout (a `PipelineEmailTemplate` that splits the body into paragraphs and auto-links any URLs) via `/api/admin/send-email`, logging the send to the project's status history; or **Copy** drops a subject + body onto the clipboard to paste into his own mail client. The recipient is always resolved server-side from the project id — never trusted from the compose form — and the route no-ops (`sent: false`) when the project has no email on file or Resend is unconfigured. The booked-elsewhere template pairs with a new terminal **`declined`** status — a prospect who chose another photographer, kept distinct from a revivable `lost` lead — which the board files under *Archived, lost & declined* and the needs-attention nudges skip.

## Project bundles

Two or more of one person's projects can be linked into a **bundle** — a wedding and its engagement session, a maternity and newborn pairing, or any grouping Julian sets up. A bundle is a shared `bundle_id` across the linked rows plus a denormalized `bundle_label` (the display name), so the relationship reads from any one project without a join. Both sides can manage it: the client links their own projects from the portal menu (`PortalBundles` → `/api/portal/bundle`), and Julian links a client's projects from the admin project page (the same component, pointed at `/api/admin/bundle`). The two write paths share one invariant — a bundle can never span two people. The client path drops any project id the signed-in email doesn't own; the admin path requires every selected project to resolve to a single email. Linking needs two or more projects and a label; unlinking clears the pair on one project.

The link is **visible by text, color, and grouping**. On the portal menu, bundled projects leave the per-category grid and render together in their own accent-bordered block headed by the `↔` glyph, the bundle label, and a "Bundle" tag; their cards carry an accent border. On both admin surfaces — the projects overview and the project detail page — a linked project shows an `↔ {label}` badge. The bundle control itself surfaces any existing bundles (label + member projects) in its collapsed header, so the grouping is visible before the panel is opened to add more or unlink. The interface matches the rest of the site: the same card shells, accent variables, and serif headings as the services / galleries menus.

## AI-assisted drafting

Every generative action across the site fires from one shared **`AiButton`** — a ✨-marked accent pill with a built-in loading state — so a generative tool always looks like one: "Help me write this" and "Draft this" on the public inquiry + questionnaire forms, plus the admin triage, prep-brief, plan, compose-email, copy-polish, journal, meta-description, alt-text, and natural-language-search tools.

The compose-email panel on a project gains an admin-only **"Draft with AI"** button that turns a pipeline template + the project's real context into a personalized, on-brand draft — which Julian reviews and edits before sending (the AI never sends; it only fills the editable subject/body). An optional one-line steer ("mention the rain backup", "warmer tone") tunes the result, which lands under an "AI draft — review before sending" note.

A provider abstraction in `src/lib/ai/ai.ts` fronts any OpenAI-compatible chat-completions API with a single `generateText({ system, prompt })`. The default is **Groq** — free, fast, and (per their policy) non-training, which is why it's the pick for prompts carrying client PII — and it's swappable to OpenAI / Together / Fireworks / any compatible endpoint via `AI_BASE_URL` + `AI_MODEL` — including Vercel's AI Gateway as a documented target (`.env.example` carries the base URL, slug format, and the caveat that enforced zero-data-retention routing is Pro-plan-gated, so direct Groq stays the default for client-data prompts). The whole module no-ops (returns null) when `GROQ_API_KEY` is unset, so the compose panel simply omits the AI button and stays a manual editor — the same optional-by-env posture as Resend and Twilio. The key is server-only, never `NEXT_PUBLIC_`.

`/api/admin/draft-email` (admin-gated, per-IP rate-limited, Sentry-instrumented) loads the project server-side and sends the model only the minimal, client-appropriate facts a draft needs — name, service, stage, date, venue, package, guest count, plan summary, and the project's portal/booking/gallery/questionnaire links — never the internal note or budget, and never PII trusted from the request body. A reusable brand-voice system prompt (`src/lib/ai/ai-voice.ts`) establishes Julian's persona and the non-negotiable rules — write as Julian, use only the supplied facts, leave a `[bracket]` placeholder rather than invent a missing detail — so a draft reads like him and can't fabricate a date or price. The model returns a `Subject:`/body pair parsed straight into the editable fields, falling back to the static filled template if the provider is unreachable. This was the first consumer of the shared `ai.ts`.

A second consumer, **inquiry triage**, reuses the same module. On a project that carries an inquiry, an admin-only **"Triage this inquiry"** button (`/api/admin/triage` — gated, per-IP rate-limited, Sentry-instrumented) reads the inquiry server-side and returns a structured assessment: a one-line summary, a fit verdict (good / possible / unlikely) with a reason, an urgency read (high / medium / low, weighing event-date proximity), the key details worth saving to the record, and a suggested first reply in Julian's voice. The model is asked for strict JSON, which a tolerant extractor (`extractJsonObject`) plus a zod schema — with `.catch()` fallbacks on the enums — validate before the UI trusts it; an unparseable response surfaces a retry rather than a broken card. Triage is **on-demand** (never auto-run on page load), so it only spends provider quota when asked, and it no-ops (`triaged: false`) without a key or an inquiry to read.

A third consumer, the **shoot-prep brief**, turns a submitted planning questionnaire into something useful on the day. On a project with a questionnaire snapshot, a **"Generate prep brief"** button (`/api/admin/prep-brief`) resolves the snapshot's field IDs back to their human labels via the questionnaire schema, groups the answers by their original sections, and asks the model for a tight, skimmable brief — a one-line headline plus sections (timeline, key people, must-have shots, locations, special moments, logistics & heads-up, style), each a few short bullets — validated through the same `extractJsonObject` + zod path. It's on-demand and no-ops (`generated: false`) without a key or a questionnaire, and the prompt is told to flag gaps as `[confirm with client]` rather than invent a time or a name.

A fourth consumer drafts the **client-facing plan summary**. The portal-visible `planSummary` field gets a **"Draft with AI"** button on the admin edit form (`/api/admin/draft-plan`); it composes a short, warm overview in Julian's voice from the client's questionnaire answers (via the same `buildAnswerDigest` the prep brief uses, factored into `src/lib/questionnaire-digest.ts`) plus neutral booking facts, and drops the result into the editable field for him to refine and save. Because the output is portal-visible, it **deliberately never reads the internal notes or budget** — only the client's own answers and facts — so a private note can't leak into a client-facing field; and it confirms before overwriting an existing summary.

A fifth consumer suggests the **next action**. A **"Suggest next step"** button on a project (`/api/admin/next-action`) reasons over the pipeline state — status, days since the last update, event proximity, and what's still missing (questionnaire, gallery, email) — and returns the single best next move with a one-line rationale, an urgency read, and, when sending is the right call, which pipeline template fits (resolved against the registry so the suggested name is real). It complements the always-on, heuristic needs-attention strip with a smarter per-project read, stays **on-demand** so the board never fans out an AI call per card, and no-ops without a key. Together these five surfaces — email drafting, inquiry triage, prep brief, plan-summary drafting, and next-action nudges — make up the AI admin layer, all on the single `ai.ts` provider abstraction (Groq by default, any OpenAI-compatible endpoint by env).

**Content tools** extend the same `ai.ts` beyond the client pipeline, gathered under a **Content** tab in the admin nav (`/admin/content`). Both draft text only and **never publish** — Julian reviews and saves in Sanity Studio.

The **journal drafter** turns a topic or brief — with an optional post type (recap, educational, personal essay, seasonal) — into a draft post in Julian's voice via `/api/admin/draft-journal`: title, excerpt, body, and suggested tags, each copyable. It's deliberately **topic-driven and never reads private client data** (the journal is public), the prompt is told not to fabricate names / prices / stats (it leaves `[brackets]` instead), and the output maps to the journal post's draftable fields — the schema-required cover image + publish date stay Julian's to add in Studio (an AI can't supply a photo).

The **copy polisher** drafts or tightens a service or portfolio page's marketing copy (`/api/admin/draft-copy`): pick a page + a mode (tighten the current copy / rewrite fresh / draft from scratch) with an optional angle, and it returns an improved tagline, description, and intro — copyable — built from **the copy that's currently live** (read server-side via `getService` / `getPortfolio`). Same guardrails: uses only the live facts, never invents prices/claims, no-ops without a key or an unresolved subject.

The **meta-description drafter** (`/api/admin/draft-meta`) writes an SEO search-result snippet (~150–160 characters, with a live character count in the UI) for one of the site's key static / index pages — home, about, the services and portfolio indexes, the journal, and inquire — from a small `src/lib/seo-pages.ts` registry of page purposes plus the site settings. These metas live in code (`metadata.description`), so the output is for review and applying to the page (service and portfolio detail pages aren't listed here — their meta derives from the on-page description the copy polisher already drafts). A shared `CopyField` component backs the copy-to-clipboard affordance across all three tools.

**Natural-language project search** adds a plain-English search box at the top of `/admin/projects` (`/api/admin/search`): a query like "weddings in June with no contract yet" is sent to the AI, which returns a **structured filter spec** — statuses, service, free text, event month/year, an upcoming-days window, and gallery / questionnaire flags — that the route then applies to the client list server-side, returning the matching projects with a note on how the query was interpreted. The semantic interpretation (e.g. "no contract yet" → the set of pre-contract statuses, "June" → the right month/year relative to today) lives in the model; the route only applies concrete, validated filters, so a malformed spec can't run arbitrary logic. It reuses the existing `listClients` read, returns only board-level fields (never internal notes), **short-circuits to no results on an empty store without calling the provider**, and no-ops without a key. The classic text/status/service filter bar stays beneath it.

**Portfolio image alt text** is generated two ways, both backed by a new vision capability in `src/lib/ai/ai.ts` — `describeImage({ system, prompt, imageDataUrl })`, the multimodal sibling of `generateText`, using a separate `AI_VISION_MODEL` (default `meta-llama/llama-4-scout-17b-16e-instruct`) so the text default is untouched. At **import time**, `npm run import-photos --alt` describes each new or still-placeholder image inline (the script makes the same Groq vision call by `fetch`, since a Node `.mjs` can't import the TS module); without the flag or a key, images keep the neutral `"<slug> photograph N"` placeholder. The importer also now **preserves** any non-placeholder alt already in the manifest across re-imports, so AI descriptions and hand edits are no longer wiped on the next run. A per-image **admin tool** (Content tools → *Portfolio image alt text*) lists a gallery's photos, generates one image's alt on demand (`/api/admin/draft-alt`, which validates the `src` belongs to the named gallery, fetches it server-side, and describes it — never publishing), and **persists** a reviewed override (`/api/admin/save-alt`). Overrides live in a small Supabase `portfolio_image_alt` table (`src` → `alt`), read through `src/lib/portfolio-alt.ts` — a self-contained, service-role module that degrades to `{}` when Supabase is unset, deliberately separate from the CRM store. The portfolio detail page overlays these overrides onto the manifest baseline at render, so a saved description wins over the import-time alt and survives the next import. The whole feature is optional-by-env at every layer: no key → placeholder alt; no Supabase → manifest alt; both absent → galleries still render.

**Public booking concierge & FAQ** complete the AI suite with its only visitor-facing surface. A new `/faq` page consolidates every question into one searchable, filterable directory — free-text search over question + answer, a collection filter (a general bucket plus each umbrella) with a per-service sub-filter, native `<details>` accordions grouped by service, and a consolidated `FAQPage` JSON-LD block. The FAQ set is a single shared source (`src/lib/faq.ts`): a curated, evergreen general set owned in code plus every visible service's `faqs[]`, each tagged with its umbrella and service so the directory can map and filter them. That same source grounds a site-wide **concierge chat** — a floating "Ask" widget (and a docked panel on `/faq`) that fields questions about services, pricing, and booking and steers toward the inquiry form. It's multi-turn (`generateText` gained an additive `messages` transcript option) and grounded only in public catalog content assembled by `src/lib/ai/concierge-kb.ts` — site settings, the condensed service catalog with prices, the shared FAQ set, and the about bio. The `POST /api/concierge` route mirrors the public inquiry pattern (per-IP rate limit, honeypot, bounded message size + history) and **never imports the client/CRM store**, so no private data can reach the model; its system prompt forbids inventing prices or availability, keeps scope to Julian's photography, ignores instructions embedded in visitor messages, and hands booking actions to the inquiry form. The model has no tools and its reply renders as plain text — nothing is executed. Everything degrades by env: the directory is always available, while the chat appears only when `GROQ_API_KEY` is set (otherwise a static pointer to the FAQ and inquiry form). The widget self-suppresses on `/faq` and on the studio / admin / portal chrome.

**Client-side writing assistant** turns the same `ai.ts` toward the people filling the forms, not just the admin reviewing them. The three open-text moments in the funnel — the inquiry's "Tell me about your vision," every questionnaire `textarea`, and the portal's "Notes / questions for me" box — share one `AssistedTextarea` component with an opt-in **"Help me write this"** affordance. It opens a two-box panel: the client jots rough notes (even keywords), the assistant shapes them into a clear, first-person draft in a _separate, editable_ box, and the field itself is touched only if they pick **Use this** — so they can refine their notes and re-draft, edit the suggestion directly, or cancel back to their own words. Each surface feeds the draft the facts already known this session — the client's other answers so far, or (in the portal) the project's service / date / status — so it reads as theirs and stays consistent, while the model is forbidden from inventing any specific (name, date, venue, price) not present in those facts. A single `POST /api/assist/draft` route mirrors the concierge's public posture (per-IP rate limit, honeypot, bounded sizes) and is **stateless** — it imports no catalog or CRM data and persists nothing, so the client PII it shapes only passes through the non-training provider. It stays a writing assistant, never the photographer: the prompt won't quote offerings or answer on Julian's behalf, treats the notes + context as material rather than instructions, and returns the draft plus an optional one-line `tip` nudging a detail worth adding when the notes are thin. The affordance appears only when `GROQ_API_KEY` is set (threaded to the client as a server-evaluated flag, since the key is server-only); without it, every field is a plain textarea.
